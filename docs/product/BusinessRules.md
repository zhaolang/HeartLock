## 1. Purpose（目的）

本文档定义 HeartLock（心锁）所有业务规则的详细规范。每一条规则都有唯一编号，供需求、设计、开发和测试环节精确引用。

---

## 2. Scope（范围）

涵盖用户账户、心锁生命周期、匹配机制、数据隐私、安全合规等方面的所有硬性业务规则。

---

## 3. Definitions（术语）

| 术语 | 定义 |
|---|---|
| 心锁 | 一份加密存储的喜欢情感数据单元 |
| 等待中（WAITING） | 已创建但尚未匹配的心锁状态 |
| 已匹配（MATCHED） | 双向喜欢已确认的状态 |
| 已撤回（REVOKED） | 用户手动取消等待的状态 |
| 已销毁（DESTROYED） | 用户永久删除的状态 |
| 手机号指纹 | 对手机号加盐哈希后得到的不可逆标识 |

---

## 4. Business Rules（业务规则）

### 4.1 账户规则

**RULE-001** 一个用户必须通过华为快捷账号登录后才能使用 App。
**RULE-002** 用户必须在登录后授权手机号，才能创建心锁（RULE-002a：未授权手机号的用户只能浏览首页和个人中心）。
**RULE-003** 一个手机号只能注册一个 HeartLock 账户。
**RULE-004** 账户注销操作需要用户二次确认（RULE-004a：首次点击弹出确认弹窗；RULE-004b：第二次点击确认后立即执行）。
**RULE-005** 账户注销后，该手机号**不可**再次注册新账户。
**RULE-006** 账户注销后，用户在所有表中的数据**必须**彻底删除，不可恢复（RULE-006a：用户表删除；RULE-006b：心锁表删除；RULE-006c：Push Token 删除；RULE-006d：操作日志保留 7 天后清除）。

### 4.2 心锁创建规则

**RULE-010** 用户向同一手机号**只能**创建一次心锁，无论该心锁当前处于何种状态（WAITING / REVOKED / DESTROYED）。
**RULE-011** 用户**最多**同时拥有 **3** 个处于 WAITING 状态的心锁（RULE-011a：当已有 3 个 WAITING 心锁时，创建新心锁被拒绝；RULE-011b：撤回或匹配成功可释放名额）。
**RULE-012** 创建心锁时必须指定目标手机号（RULE-012a：手机号仅限中国大陆手机号格式；RULE-012b：手机号不能是自己的手机号）。
**RULE-013** 创建心锁时必须填写文字内容（RULE-013a：最少 1 个字符；RULE-013b：最多 500 个字符；RULE-013c：内容在写入数据库前进行 AES-256 加密）。
**RULE-014** 创建心锁时，服务器对用户手机号和目标手机号分别进行加盐哈希，将指纹存入数据库（RULE-014a：明文手机号不落盘）。

### 4.3 心锁状态转换规则

**RULE-020** 心锁状态转换图：`WAITING → MATCHED` 或 `WAITING → REVOKED → DESTROYED`
**RULE-021** 处于 MATCHED 状态的心锁不可撤回或删除。
**RULE-022** 处于 DESTROYED 状态的心锁不可转回其他任何状态。
**RULE-023** 撤回操作（WAITING → REVOKED）时，保留所有元数据（创建时间、双方指纹），仅标记状态变更（RULE-023a：加密内容保留但不参与匹配检测）。
**RULE-024** 永久删除操作（REVOKED → DESTROYED）时，将心锁内容字段置为 NULL，彻底不可恢复（RULE-024a：元数据继续保留 30 天用于防重复检测，30 天后清除）。

### 4.4 匹配检测规则

**RULE-030** 每次创建心锁时，服务器立即进行匹配检测。
**RULE-031** 匹配检测逻辑：查找是否存在一条心锁满足（RULE-031a：目标手机号指纹 = 当前用户手机号指纹 **且** RULE-031b：发送者手机号指纹 = 当前用户创建心锁的目标手机号指纹 **且** RULE-031c：该心锁状态为 WAITING）。
**RULE-032** 匹配成功后，两条心锁**同时**将状态更新为 MATCHED（RULE-032a：两条记录的 matched_at 时间戳相同）。
**RULE-033** 匹配检测必须在 2 秒内完成。

### 4.5 通知规则

**RULE-040** 系统仅在匹配成功时发送 Push 通知。
**RULE-041** 匹配通知内容：仅显示"心锁已打开"和对方写给自己的第一句话（前 50 字）。
**RULE-042** 禁止在任何情况下向用户发送以下内容的通知（RULE-042a：有人喜欢你；RULE-042b：有人收藏了你；RULE-042c：有一封给你的信；RULE-042d：有人在等你）。
**RULE-043** 禁止在 App 内任何界面显示以下内容的提示（RULE-043a：红点/角标暗示有心锁动态；RULE-043b：首页等待时长的暗示性文案）。

### 4.6 数据安全规则

**RULE-050** 心锁文字内容使用 AES-256-GCM 加密存储。
**RULE-051** 加密密钥与数据分离存储（RULE-051a：密钥存储在独立的密钥管理服务中）。
**RULE-052** 手机号使用 bcrypt（cost=12）+ 随机 salt 进行哈希存储。
**RULE-053** 所有 API 请求必须使用 TLS 1.3。
**RULE-054** 服务器响应中**不得**返回任何形式的手机号明文。
**RULE-055** 所有数据库备份数据也必须加密存储。

### 4.7 邀请裂变规则

**RULE-060** 用户创建心锁后，可在心锁详情页生成邀请卡片。
**RULE-061** 邀请卡片不得包含以下内容（RULE-061a：发送者的任何身份信息，包括昵称/头像/手机号指纹；RULE-061b：发送者的文字内容；RULE-061c：任何暗示"有人在等你"的文案）。
**RULE-062** 邀请卡片必须包含"打开心锁"按钮，点击后跳转到应用市场下载 HeartLock。
**RULE-063** 同一心锁的邀请卡片不可重复生成（每个心锁仅可生成一张卡片）。

## 5. Acceptance Criteria（验收标准）

| 编号 | 验收标准 | 关联规则 |
|---|---|---|
| AC-BR-001 | 同一手机号向同一目标手机号第二次创建心锁时，服务端拒绝并返回错误 | RULE-010 |
| AC-BR-002 | 同时有 3 个 WAITING 心锁时，第 4 次创建被拒绝 | RULE-011 |
| AC-BR-003 | 创建心锁时，目标手机号不能为创建者自己的手机号 | RULE-012b |
| AC-BR-004 | 双方互相创建心锁后 2 秒内，状态同时变为 MATCHED | RULE-030 ~ RULE-033 |
| AC-BR-005 | 匹配成功后双方同时收到 Push，内容仅为解锁通知和对方第一句话 | RULE-040 ~ RULE-041 |
| AC-BR-006 | 未匹配时，App 不显示任何暗示性提示或红点 | RULE-042 ~ RULE-043 |
| AC-BR-007 | 账户注销后查询对应手机号的任何心锁记录返回空 | RULE-005 ~ RULE-006 |
| AC-BR-008 | 邀请卡片不包含任何创建者身份信息 | RULE-060 ~ RULE-063 |
